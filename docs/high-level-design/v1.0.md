# Cardano Stake Pool 控制平面 - 概要设计文档 (High-Level Design)

**项目代号：** Cardano-SPO-Pilot (M0)  
**文档版本：** v1.0  
**创建日期：** 2026-03-02  
**核心目标：** 打造 macOS 原生、安全、向导式的 Cardano 矿池运维工具。

---

## 一、 架构原则 (Architecture Principles)

1.  **Push 模式运维：** 本地作为控制中心，通过 Ansible 远程推送指令，服务器端零 Agent 安装。
2.  **Sidecar 隔离执行：** 将 Python 与 Ansible 环境封装为 App 的 **Sidecar**（边车进程），确保用户环境无污染。
3.  **密钥硬隔离：**
    * **SSH Key：** 托管于 macOS Keychain，通过 SSH Agent 调用，应用内存不留私钥。
    * **Cold Key：** 物理隔离，应用仅处理签名请求 (Sign Request) 与证书导入。
4.  **状态最终一致性：** 每次任务执行前后进行环境幂等性检查，确保集群状态符合预期。

## 三、 核心模块详细设计

### 3.1 主机管理模块 (Machine Manager)
* **SSH 代理集成：** 应用启动时自动连接系统 `ssh-agent`。
* **权限校验：** 添加主机时，通过 `ssh-add -l` 校验对应指纹是否已加载，若未加载则引导用户通过 Keychain 授权。
* **动态 Inventory：** 执行任务时，从 SQLite 提取数据实时生成临时内存 Inventory，避免在磁盘留下明文主机列表。

### 3.2 部署引擎 (Deployment Engine)
* **环境初始化：**
    * **时间同步：** 自动部署 `Chrony`，强制校准系统时间。
    * **内存优化：** 自动检测物理内存，若 < 16GB 则自动配置 8GB-16GB 的 **Swap 交换空间**。
* **安全加固：** 默认执行 `UFW` 配置，BP 节点仅允许受信任的 Relay IP 访问 3001 端口。
* **同步检测：** 封装 `cardano-cli query tip`，计算区块同步百分比。

### 3.3 升级引擎 (Upgrade Engine)
* **滚动更新逻辑：**
    1.  **备份：** 备份 `/db` 以外的所有配置文件。
    2.  **顺序执行：** `Relay_1` -> `Relay_2` -> ... -> `Relay_N` -> `BP`。
    3.  **健康门禁：** 每台机器重启后，必须通过 `SyncProgress == 100%` 校验方可继续下一台。
* **回滚：** 任务失败时提供 `Rollback` 按钮，通过容器镜像版本回填实现快速恢复。

### 3.4 KES 自动管理逻辑
* **预警流程：** 轮询各节点 `/metrics` 接口，解析 `kesPeriod` 指标。
* **离线签名工作流：**
    * **Step 1:** App 产生新 KES 密钥对，导出签名请求包。
    * **Step 2:** 用户在离线环境使用 Cold Key 签署证书。
    * **Step 3:** App 将证书推送至 BP，执行 `SIGHUP` 信号平滑重载配置，无需完全重启容器。

## 四、 核心流程设计 (Mermaid)

### 4.1 滚动升级时序图
```mermaid
sequenceDiagram
    participant UI as macOS UI
    participant Rust as Rust Core
    participant Runner as Ansible Runner
    participant Node as Remote Node

    Rust->>Runner: 启动升级 Playbook (Serial Mode)

        Node-->>Runner: 100% 同步成功

    Runner-->>UI: Relay 升级完毕，请求 BP 升级确认
    UI->>Runner: 用户确认升级 BP
    Runner->>Node: 更新 BP 节点
    Runner-->>UI: 升级任务全部完成

## 五、 数据模型 (Data Model)

### 5.1 核心表结构 (SQLite)

```sql
-- 主机信息
CREATE TABLE machines (
    id INTEGER PRIMARY KEY,
    name TEXT NOT NULL,
    ip TEXT NOT NULL,
    role TEXT CHECK(role IN ('relay', 'bp', 'archive')),
    network TEXT CHECK(network IN ('mainnet', 'testnet')),
    ssh_fingerprint TEXT,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 任务记录
CREATE TABLE task_history (
    id INTEGER PRIMARY KEY,
    task_type TEXT, -- 'deploy', 'upgrade', 'kes_rotation'
    status TEXT,    -- 'running', 'success', 'failed'
    log_path TEXT,
    start_time TIMESTAMP,
    end_time TIMESTAMP
);

-- 矿池全局配置
CREATE TABLE pool_config (
    id INTEGER PRIMARY KEY,
    ticker TEXT,
    kes_expiry_date DATE,
    current_version TEXT
);
```

## 六、 安全加固设计 (Security Layer)

**Keychain 访问策略**： 应用仅申请 kSecClassGenericPassword 权限，遵循 macOS 沙盒安全限制。

**SSH 配置保护**： 自动生成的 SSH 配置文件中开启 ControlMaster auto，减少频繁握手，且仅在任务期间有效。

**二次确认逻辑**： 涉及修改 BP 节点容器状态的操作，UI 必须弹出确认框并展示执行风险提示。

## 七、 异常处理 (Error Handling)

SSH 连接超时： Ansible Runner 捕获 UNREACHABLE 错误，前端显示“网络诊断”建议。

升级中的同步停滞： 监控发现 10 分钟高度未增长，立即暂停升级流，并发送 macOS 系统通知提醒。

磁盘空间不足： 部署前执行 df -h 预检，不足 50GB 强行拦截并报错。

##八、 开发路线图 (MVP 阶段)

Phase 1: 搭建 Tauri 项目框架，集成 Python/Ansible 静态二进制文件（Sidecar）。

Phase 2: 完成主机管理 UI 与 SSH Agent 连接逻辑。

Phase 3: 编写部署 Playbook（含 Docker、Swap、Chrony）。

Phase 4: 实现 KES 剩余时间监控与离线文件导入引导。
