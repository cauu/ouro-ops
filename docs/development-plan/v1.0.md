## 一、计划说明

### 1.1 原则

- **分阶段交付**：每阶段结束有可运行、可验证的交付物。
- **BDD 驱动**：用 Given/When/Then 描述行为，阶段验收以场景通过为准。
- **测试门禁**：阶段完成前，该阶段对应的测试用例必须通过（见各阶段「验证」）。
- **Todo 进度**：使用下方可勾选列表记录完成情况，`[ ]` 未完成，`[x]` 已完成。

### 1.2 如何使用本 Todo 列表

- 在编辑器中直接勾选已完成项（将 `[ ]` 改为 `[x]`）。
- 每完成一项，运行对应测试或手工验证后再勾选。
- 阶段内所有 Todo 勾选完毕且「验证」中列出的测试通过后，该阶段视为完成。

## 二、Phase 1：基础框架 (Week 1-2)

### 2.1 阶段目标

应用可启动，具备 SQLite 持久化、Sidecar 进程管理与 Tauri 事件通道，为后续主机管理与部署提供基础设施。

### 2.2 BDD Feature：基础设施就绪

**Feature: 应用启动与数据持久化**

```gherkin
Scenario: 首次启动时数据库迁移成功
  Given 应用首次安装或数据目录为空
  When 用户启动应用
  Then 执行 SQLite 迁移 (001_init)
  And pool / machine / task / task_machine / machine_health / audit_log 表存在
  And user_version 正确递增
  And 应用不崩溃

Scenario: 再次启动时迁移幂等
  Given 数据库已存在且 user_version 为 N
  When 用户再次启动应用
  Then 仅执行版本号 > N 的迁移
  And 已有数据不被破坏
```

**Feature: Sidecar 进程通信**

```gherkin
Scenario: 应用启动后 Sidecar 就绪
  Given 应用已启动
  When Rust 向 Sidecar 发送 ping
  Then Sidecar 在合理时间内返回成功
  And Rust 侧认为 Sidecar 可用

Scenario: 执行 Playbook 时收到事件流
  Given Sidecar 已就绪且存在可执行的 playbook
  When Rust 发送 run_playbook 请求（含 inventory 与 extra_vars）
  Then Sidecar 通过 stdout 逐行输出 runner_on_ok / runner_on_failed 等事件
  And 最后输出 playbook_complete
  And Rust 能解析每行 JSON 并转发为 Tauri 事件

Scenario: 应用退出时 Sidecar 优雅关闭
  Given Sidecar 正在运行
  When Rust 发送 shutdown
  Then Sidecar 进程在 5 秒内退出
  And 若未退出则 Rust 强制 kill
```

**Feature: Tauri 事件通道**

```gherkin
Scenario: 长任务可向前端推送进度
  Given 后端启动了一项会产生进度的任务
  When Sidecar 上报 task 完成或日志行
  Then Rust 发出 task:progress 或 task:log 事件
  And 前端可通过 listen 收到 payload（含 task_id、machine_name、line 等）
```

## 三、Phase 2：主机管理与矿池 (Week 3-4)

### 3.1 阶段目标

用户可完成矿池初始化（单池）、主机增删查、SSH 连通性测试与环境预检；前端具备路由与主机/矿池相关页面。

### 3.2 BDD Feature：矿池与主机管理

**Feature: 矿池初始化（单池）**

```gherkin
Scenario: 首次启动时无池则进入初始化向导
  Given 应用启动且 pool 表为空
  When 前端调用 pool_get()
  Then 返回错误或空
  And 前端路由跳转至 /setup

Scenario: 用户在向导中完成矿池初始化
  Given 用户在 /setup 页面
  When 输入 ticker（3～5 字符）、选择 network、可选 margin/fixed_cost 并提交
  Then 调用 pool_init(payload)
  And 数据库插入一条 pool 记录
  And 返回 Pool 对象
  And 前端跳转至 /

Scenario: 已存在池时禁止再次初始化
  Given 数据库中已有一条 pool
  When 调用 pool_init(任意 payload)
  Then 返回错误
  And pool 表仍只有一条记录
```

**Feature: 主机 CRUD 与 SSH**

```gherkin
Scenario: 添加主机时校验 SSH 并写入数据库
  Given 已存在 Pool，ssh-agent 中有所选密钥
  When 用户提交 machine_add(name, ip, port, ssh_user, role, network, ssh_key_fingerprint)
  Then 后端校验 ssh_key_fingerprint 在 agent 中存在
  And 尝试 SSH 连接目标主机
  And 成功则插入 machine 表并写 audit_log
  And 返回 Machine 对象

Scenario: 同 Pool 下同一 IP 不可重复添加
  Given 已存在一台 machine 的 ip 为 203.0.113.10
  When 再次 machine_add 同一 ip（同一 pool）
  Then 返回错误
  And UNIQUE(pool_id, ip) 不被违反

Scenario: 预检返回磁盘与内存信息
  Given 存在一台 SSH 可达的 machine
  When 调用 machine_preflight(machine_id)
  Then 返回 PreflightReport：ssh_ok、os_version、disk_available_gb、memory_total_gb、warnings
  And 若磁盘 < 300GB 则在 warnings 中提示
```

## 四、Phase 3：部署引擎 (Week 5-7)

### 4.1 阶段目标

用户可通过部署向导选择机器与参数、启动部署任务；后端按设计生成 Inventory、执行 deploy playbook，前端展示实时日志与进度；支持取消与状态查询。

### 4.2 BDD Feature：部署流程

**Feature: 部署任务生命周期**

```gherkin
Scenario: 用户完成向导后启动部署
  Given 已存在 Pool 与多台 machine（至少 1 relay）
  When 用户在 Deploy 向导中选择 machine_ids、cardano_version、network、swap/chrony/hardening 并确认
  Then 调用 deploy_start(payload)
  And 后端执行 PREFLIGHT（可选预检）
  And 生成 inventory（relay/bp 分组、hostvars）
  And 调用 Sidecar run_playbook(deploy.yml, inventory, extra_vars)
  And 返回 TaskId
  And 前端展示任务与 TaskLogStream

Scenario: 部署过程中前端收到实时日志
  Given 部署任务已启动
  When Sidecar 输出 Ansible 的 stdout/stderr 行
  Then 前端收到 task:log 事件（task_id、stream、line、timestamp）
  And TaskLogStream 仅显示当前 task_id 的日志

Scenario: 用户取消运行中的部署
  Given 存在 status=running 的 deploy task
  When 调用 deploy_cancel(task_id)
  Then 任务状态变为 cancelled
  And Sidecar 当前 play 被终止
  And 前端可收到 task:failed 或等价结束事件
```

**Feature: Inventory 与 Playbook**

```gherkin
Scenario: Inventory 包含 relay/bp 分组与 hostvars
  Given 当前 Pool 下有 2 relay、1 bp
  When 为 deploy 构建 inventory(machine_ids)
  Then 生成 JSON 含 _meta.hostvars（每台 ansible_host、ansible_port、role、network、relay_nodes、bp_nodes、trusted_relay_ips）
  And 含 "relay": {"hosts": [...]}, "bp": {"hosts": [...]}
```

## 五、Phase 4：监控与 KES、升级 (Week 8-10)

### 5.1 阶段目标

Dashboard 展示节点状态卡片与 KES 倒计时；监控轮询与告警阈值生效；KES 生成/导入证书并推送 BP；升级与回滚流程可用，门控与二次确认符合设计。

### 5.2 BDD Feature：监控与 Dashboard

**Feature: 监控数据与告警**

```gherkin
Scenario: 获取所有节点指标快照
  Given 存在多台 machine
  When 调用 monitor_snapshot()
  Then 返回 Vec<NodeMetrics>（block_height、sync_progress、peers、cpu/mem/disk、status）
  And 通过 SSH 或已有连接采集 Prometheus 指标与系统指标

Scenario: 轮询时向前端推送更新
  Given 已调用 monitor_start_polling(interval_secs)
  When 到达轮询周期
  Then 采集各节点指标并写入 machine_health
  And 发出 monitor:update 事件（Vec<NodeMetrics>）
  And status 按阈值设为 healthy/warning/critical
```

**Feature: KES 生命周期**

```gherkin
Scenario: 生成 KES 密钥对并返回签名指引
  Given machine_id 为 BP
  When 调用 kes_generate(machine_id)
  Then 本地调用 cardano-cli 生成 kes.skey/kes.vkey
  And 返回 KesSignRequest（kes_vkey_path、counter_value、instructions）

Scenario: 导入有效证书后推送并重载 BP
  Given 用户已在离线机签名得到 node.cert
  When 调用 kes_import_cert(machine_id, cert_path)
  And 校验证书格式与 counter
  Then 执行 kes-push playbook 推送 kes.skey + node.cert 至 BP
  And BP 收到 SIGHUP 平滑重载
  And 更新 kes_state 表
  And 发出 task:completed
```

**Feature: 滚动升级与回滚**

```gherkin
Scenario: 升级至门控时等待用户确认
  Given 滚动升级已执行完一台 Relay 或全部 Relay
  When 需要继续下一台或升级 BP
  Then 发出 upgrade:gate（completed_machine、next_machine、is_bp）
  And 仅当用户调用 upgrade_confirm_next 后继续执行

Scenario: 升级 BP 前必须高危确认
  Given 当前处于 AWAIT_BP_CONFIRM
  When 前端展示确认 Modal
  Then Modal 为 dangerous 级别，需输入 Ticker 才能确认
  And 确认后调用 upgrade_confirm_next

Scenario: 回滚恢复备份与旧镜像
  Given 升级任务已备份且 payload 含 previous_version 与 backup_archive
  When 调用 upgrade_rollback(task_id, machine_id)
  Then 执行 rollback.yml：恢复配置、旧镜像启动、健康检查
  And 成功后更新 machine.cardano_version
```
