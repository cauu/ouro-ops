# Cardano Stake Pool 控制平面 (macOS) 需求文档

**版本：** v1.1 **状态：** 详细方案定稿 **定位：** 独立开发者与 SPO 的 macOS 原生运维工具，旨在实现“无命令行为”的矿池生命周期管理。

---

## 1. 核心设计原则

- **本地驱动 (Local-First):** 采用 Ansible **Push 模式**。远程服务器仅需 Python 环境与 SSH 权限，无需安装管理代理。
    
- **安全隔离 (Security-Centric):** 所有 SSH Private Keys 存于 macOS Keychain；**Cold Keys 永远保持物理离线**。
    
- **向导式运维 (Guided Ops):** 将复杂的 Cardano 运维指令抽象为可视化向导，杜绝因误操作导致的丢块。
    

---

## 2. 核心功能模块

### 2.1 主机管理模块 (Machine Manager)

**目标：** 实现多节点集群的拓扑管理。

- **节点录入：** 支持 IP、SSH 用户、端口及 SSH Key 指向。
    
- **自动标签：**
    
    - **角色标签：** `Relay` / `Block Producer` / `Archive`。
        
    - **网络标签：** `Mainnet` / `Preprod` / `Preview`。
        
- **主机加固 (Hardening):** 部署后自动禁用远程密码登录，强制 SSH Key 验证，并配置 `Fail2ban`。
    
- **环境预检：** 自动检测 OS 版本、磁盘 I/O 性能、剩余空间（建议 >300GB）及内存容量。
    

### 2.2 部署引擎 (Deployment Engine)

**目标：** 一键完成从零到同步完成的节点搭建。

- **系统环境优化：**
    
    - **时间同步：** 自动安装配置 `Chrony/NTP`，确保时间偏差 < 200ms。
        
    - **虚拟内存：** 自动创建 8GB-16GB 的 **Swap 文件**，防止节点同步导致的 OOM。
        
    - **参数调优：** 自动设置 `ulimit` 及 Docker 日志轮转策略（防止磁盘被日志撑爆）。
        
- **拓扑自动化：**
    
    - 自动根据节点角色生成 `topology.json`。
        
    - 配置 BP 仅允许与受信任的 Relay 通信（Firewall 自动放行规则）。
        
- **状态追踪：** 实时显示容器拉取、目录创建及区块同步进度（Sync Percentage）。
    

### 2.3 升级引擎 (Upgrade Engine)

**目标：** 无损、可回滚的滚动升级。

- **滚动更新流：** 强制执行“先 Relay，后 BP”的升级顺序。
    
- **升级验证：**
    
    1. 升级首台 Relay。
        
    2. 健康检查：确认 `SyncProgress` 恢复至 100%。
        
    3. 确认无误后方可点击“继续升级下一台”。
        
- **快照与回滚：** 升级前自动备份配置目录；若新版本启动失败（例如进程在 5 分钟内崩溃），提供一键切回旧镜像的选项。
    

### 2.4 KES 生命周期管理 (KES Manager)

**目标：** 解决 SPO 最核心的痛点——KES 过期导致的停产。

- **监控预警：** 实时显示 KES 剩余天数，< 10 天触发橙色预警，< 3 天触发红色告警。
    
- **离线签名工作流 (Air-gapped Flow)：**
    
    1. **发起请求：** App 在本地生成新 KES 密钥对。
        
    2. **导出文件：** 导出需要 Cold Key 签名的 `kes.vkey`。
        
    3. **用户操作：** 用户在离线机器签名生成 `node.cert`。
        
    4. **导入与推送：** 用户将证书拖入 App，App 自动通过 Ansible 推送至 BP 并平滑重启节点。
        

### 2.5 监控与告警模块 (Monitoring Engine)

**目标：** 实时感知矿池健康度。

- **关键指标看板：**
    
    - **核心状态：** 区块高度、Peers 连接数、Mempool 大小。
        
    - **资源占用：** CPU、内存、磁盘负载。
        
    - **业务指标：** **Missed Slots (漏块检测)**、KES 剩余周期、池 Margin/Cost 变更记录。
        
- **状态分类：** `Healthy` (绿色) / `Warning` (黄色) / `Critical` (红色)。

## 3. 技术架构与安全逻辑

### 3.1 安全控制层

- **Keychain 集成：** 所有敏感的 SSH 路径和密码不以明文形式出现在配置文件中。
    
- **确认机制：** 涉及修改链上状态或重启 BP 的操作，必须进行二次确认（Modal 弹窗）。
    
- **审计日志：** 本地记录所有执行过的 Ansible 指令及其结果 JSON，方便追溯。
    

### 3.2 数据模型 (Core Entities)

SQL

```
-- 节点信息
Machine {
  id, name, ip, role, network, 
  os_version, ssh_key_ref, 
  last_health_check_time
}

-- 矿池配置
Pool {
  id, ticker, network, 
  margin, fixed_cost, 
  kes_expiry_date, 
  relay_list: Machine[], 
  bp_node: Machine
}
```

## 4. MVP (最小可行性产品) 范围

- **第一阶段：** 支持多主机添加与 SSH 通信测试。
    
- **第二阶段：** 完成基于 Docker 的向导式部署（含时间同步与 Swap 优化）。
    
- **第三阶段：** 实现基础监控 Dashboard（Sync %、KES 倒计时）。
    
- **第四阶段：** 完善 KES 更新流程（手动导入证书模式）。
    

---

## 5. 成功标准

1. **零命令操作：** 用户无需手动打开 Terminal 即可完成池的完整搭建。
    
2. **安全基准：** BP 节点在默认配置下不暴露任何公网端口，且仅允许来自白名单 Relay 的流量。
    
3. **高可用：** 升级过程中，矿池始终保持至少一个 Relay 处于在线同步状态。
